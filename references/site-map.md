# 业务线 + 卡片家族体系（v0.5）

> **本文件定义**：
> - `business_line` 枚举（页面/流级，人工声明）
> - `card_type` 枚举（单卡级，Skill 视觉识别）
> - 家族识别决策树（Skill 执行的核心逻辑）

---

## 一、business_line 枚举（人工声明）

调用 Skill 时，调用方必须从以下枚举选取：

| business_line | 核心场景定义 |
|---|---|
| **推荐** | 通用内容发现流（精选 / 兴趣匹配 / PGC+UGC 混合）|
| **点评** | 商品评价消费、UGC 测评内容 |
| **圈子** | 社区内容、兴趣社群、话题流 |
| **新品** | 新品种草、上新发现、发布事件 |
| **个人中心** | 个人主页、收藏、关注、我的发布、种草 |
| **直播** | 直播广场、品类直播、直播推荐流 |

**关键协议**：`business_line` 由调用方**显式声明**，Skill **不推断**。原因：页面内 tab 命名会变（"JoyAI""求真官"等易变信号），用可变信号做分类不稳定。

---

## 二、card_type 枚举（Skill 视觉识别）

Skill 对每张卡独立识别 `card_type`。结构为 **{家族, 子类型, 媒介}** 三元组。

### 2.1 五大家族（顶层分类）

| family | 心理锚点 | 底部 DNA（L1-5 家族识别核心）|
|---|---|---|
| `内容` | UGC 信任 + 社交证明 | 头像 + 昵称 + 互动数据 |
| `商品` | 商家背书 + 销量证明 | 店铺 logo + 店铺名 + 销售数据 |
| `圈子` | 集体归属感 | 圈子图标 + 圈子名（无作者无互动） |
| `点评` | UGC 信任 + 决策证明 | 头像 + 昵称 + 互动 + 标题下种草数 |
| `新品` | 事件 / 活动 / 趋势驱动 | 按子类型变化（CTA/事件胶囊/挂载/话题）|
| `未知` | —— | 识别失败的兜底类别 |

### 2.2 子类型（按家族分组）

详见 `business-baselines.md` 第三节。这里列出完整枚举：

```yaml
card_subtype_enum:
  内容:
    - 内容-图文卡  # 基础，互动可切换（点赞/观看/评论/收藏）
    - 内容-视频卡
    - 内容-直播卡·基础
    - 内容-直播卡·红包雨变体
    - 内容-直播卡·超头/达人变体
    - 内容-直播卡·预约变体
    - 内容-直播卡·沉浸式变体

  商品:
    - 商品-图文卡  # 基线待补
    - 商品-视频卡
    - 商品-直播卡·基础
    - 商品-直播卡·弹幕外露决策信息变体
    - 商品-直播卡·沉浸式商卡变体
    - 商品-直播卡·沉浸式促销变体（与内容·达人沉浸式形成翻转对照）

  圈子:
    - 圈子-图文卡
    - 圈子-视频卡  # 按规律预测，待确认

  点评:
    - 点评-图文卡
    - 点评-视频卡
    - 点评-图文卡·用户身份标签变体
    - 种草-图文卡（个人页混血，归点评家族附近）
    - 种草-视频卡（同上）

  新品:
    - 新品-入口卡
    - 新品-直播卡入口
    - 新品-测评视频卡
    - 新品-趋势专属图文卡
    - 新品-明星同款卡

  未分类:
    - 运营位  # 不在 Skill 审查范围，切分时排除
    - 广告位  # 同上
    - 未知家族  # 识别失败时的标记
```

## 三、家族识别决策树（★ Skill 执行的核心，v0.5.2 修订）

**v0.5.2 关键修订**：家族识别只依赖**结构不变量**（固定元素），不依赖**动态数据**（种草数 / 观看量等，运行时可能为 0 → 不显示）。

**分两层判断**：
- Layer 1：看底部基础结构，分"结构簇"
- Layer 2：结构相似的家族（内容 vs 点评）靠 `business_line` 辅助区分

```
STEP 2a: 看底部基础结构

CASE A: 底部 = 头像 + 昵称 + 互动数据
    # 内容家族和点评家族的底部基础结构相同
    # 不能用"有无种草数"判定，种草数是动态数据
    IF business_line ∈ {点评, 个人中心}:
        family = 点评
    ELSE IF business_line = 推荐:
        family = 内容
    ELSE:
        family = 内容/点评（模糊，标 低置信度）

    # 附加元素作为"激活信号"记录，不影响家族判定
    annotations:
      - has_seed_count: 标题下方是否有"X 人种草"
      - has_identity_badge: 昵称旁是否有身份胶囊（如"求真官"）
      - interaction_icon: 右下图标是 ❤️/👁/💬/⭐

CASE B: 底部 = 店铺 logo + 店铺名 + 价格/销量
    family = 商品

CASE C: 底部 = 圈子图标 + 圈子名（右下 = 空，无互动）
    family = 圈子

CASE D: 底部 contains [CTA按钮 / 事件胶囊 / 白底商品挂载浮条(含商品名+描述) / 话题标签+热度数字]
        AND 无作者 AND 无店铺 AND 无圈子
    family = 新品

ELSE:
    family = 未知

STEP 2b: 看封面+标题 → 识别子类型
STEP 2c: 看左上角 → 识别 media

SPECIAL CASES:
  种草卡（左上话题标签 + 封面下商品挂载 + 底部 UGC）→ family = 点评（种草子分支）
  明星同款卡（封面明星 + 底部商品挂载 + 无作者）→ family = 新品（隐形作者）
```

### 动态数据的正确处理

| 元素 | 性质 | 是否参与家族识别 |
|---|---|---|
| 头像 + 昵称 | 结构 | ✓ 是 |
| 互动数据位（有图标和占位）| 结构 | ✓ 是 |
| 店铺 logo + 店铺名 | 结构 | ✓ 是 |
| 圈子图标 + 圈子名 | 结构 | ✓ 是 |
| 互动数据的**数字**（313 / 700）| 动态 | ❌ 不参与（数字多大不影响家族）|
| **种草数的有无** | 动态 | ❌ 不参与（0 人种草 = 不显示）|
| **观看量的有无**/大小 | 动态 | ❌ 不参与 |
| 身份标签（求真官等）| 结构 | ✓ 参与（独有信号，家族锚点）|

**简单记忆**：**固定位、固定结构 → 参与识别；可能为 0、可能消失 → 不参与**。

### 3.1 识别置信度

| 条件 | 置信度 |
|---|---|
| 底部特征 + 封面特征 + 左上角 3 项都匹配某家族的典型模式 | 高 |
| 底部明确但子类型特征模糊 | 中 |
| 底部模糊或部分遮挡 | 低 |
| 完全无法匹配 | 家族 = `未知` |

低置信度时在报告里标注，**不强行归类**。

## 四、调用协议

### 输入
见 `prompts/page-level-review.md` 第一节。核心：
```yaml
business_line: <enum>  # 必填
page_screenshot: <path>  # 必填
page_name: <string>  # 必填
```

### 关键协议
- ❌ Skill 不根据页面 tab / URL / 封面特征推断 business_line
- ✅ Skill 对截图中每张卡独立识别 card_type（家族 + 子类型 + 媒介）
- ✅ 允许"一个 business_line 里混多种 card_type"（比如"直播"页里除直播卡还有商品卡/运营位）
- ✅ 若某卡 card_type 的家族与 business_line 预期不匹配 → **作为流级观察项，不强行修正**

## 六、业务线预期家族对照表（用于 Step 3 检查）

调用 `business_line=X` 时，Skill 参考下表判断"本页出现的家族是否合理"：

| business_line | 主体家族 | 允许的其他家族 | 非预期家族（值得讨论）|
|---|---|---|---|
| 推荐 | 内容 | 少量商品/新品/点评 | 圈子 |
| 点评 | 点评 | 内容 | 商品/圈子/新品 |
| 圈子 | 圈子 | — | 其他都算非预期 |
| 新品 | 新品 | 少量内容/商品 | 圈子/点评 |
| 个人中心 | 点评（种草变体）/内容 | 圈子/新品 | 商品 |
| 直播 | 内容（直播）/ 商品（直播） | 运营位（不纳入审查）| 圈子/新品入口卡（非直播形式） |

**发现非预期家族时的处理**：
1. 不作违规判决
2. 纳入报告"流级观察"项
3. 提供"可能的业务理由"（如分发算法异常、有意混排测试等）
