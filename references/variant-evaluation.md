# 变体评估协议 ★

> **本文件是本 Skill 的差异化核心**。其他文件解决"合规不合规"，本文件解决"**创新值不值**"。
>
> 行业内（包括京东 UED 公开资料、小红书/抖音/Instagram 的公开设计文档）都**没有系统化地回答过这个问题**。这是本专项最能立住的环节。

---

## 一、为什么需要变体评估协议

### 问题
多业务线做双列卡创新时，"设计师因为业绩/展示/个人表达欲做出来的创新"和"真正给用户创造价值的创新"往往**难以区分**。没有机制，评审就变成"谁声音大谁赢"。

### 本协议的目标
提供一个**简单、刚性、不讲情面**的裁决标准，让：
- 设计师**提出**创新时知道举证门槛在哪
- 评审者**裁决**创新时有统一标准
- 规范 owner **决定**是否将创新纳入下个版本

---

## 二、核心判定原则：跨场景同物种识别

### 单一裁决标准

> **一张双列卡从场景 A 搬到场景 B，用户能否在 0.5 秒内无障碍识别它是同一物种？**
>
> - **能** → 通过（即便偏离了细则）
> - **不能** → 认知负担，拒绝或升级为正式分支

**这条标准比任何细则更抗不讲理**。因为：
- 不依赖"设计师自我感觉良好"
- 不依赖业务方话术
- 不依赖规范文档的文字游戏
- 唯一考题是：**用户能否识别**

### 为什么是 0.5 秒

参考用户眼动研究的"首瞥识别"时间窗口。双列流是高速浏览场景，用户在一张卡上停留的平均时间通常 < 1 秒，留给"这是什么物种"的判断时间约 0.3~0.5 秒。超过这个时间，用户会跳过或产生认知负担。

### 为什么是"同物种"不是"同款"

- **同款**：一模一样的卡片 —— 标准太严，扼杀合理创新（比如直播卡加"直播中"标签）
- **同物种**：核心识别信号一致（结构、层级、视觉调性），局部差异可接受 —— 既允许场景化适配，又约束创意膨胀

### 识别信号的优先级

用户识别"这是一张双列内容卡"靠的是（从重要到次要）：

| # | 识别信号 | 识别阈值 |
|---|---|---|
| 1 | 宽度、间距、圆角 | 和标准卡偏差 < 10% |
| 2 | 封面占主体（≥ 60%） | 封面必须是视觉主角 |
| 3 | 标题在封面下方 | 信息层级不能颠倒 |
| 4 | 有作者信息 | 社交信任锚点 |
| 5 | 互动数据/标签位置 | 次级信号，有容差 |

**前 4 条破坏任何一条 → 同物种识别失败 → 拒绝变体**。

## 三、变体评估的四个维度

| 维度 | 标准卡 | 创新卡示例 | 评估要问 |
|---|---|---|---|
| **信息元素** | 封面+标题+作者+互动 | 新增元素（AI 摘要、特殊标签、价格标） | 新元素解决了什么具体用户问题？不加有什么损失？ |
| **布局结构** | 3:4 / 9:16 | 混排、大小卡、异形 | 新布局和标准卡共存会不会打断浏览节奏？ |
| **交互** | 点击进详情 | 卡内浮层、直接下单、轻交互 | 新交互的学习成本，是否有清晰的触发信号？ |
| **视觉** | 圆角 12、间距 8 | 自定义配色、特殊阴影、动效 | 是"区分场景"还是"个人表达欲"？ |

**评估时逐维度问**。任何一个维度给出"不知道"或"感觉会更好"这种模糊答案 → 当场拒绝。

## 四、评估打分表（满分 100）

| 项 | 分值 | 评分标准 |
|---|---|---|
| **跨场景同物种识别**（★ 一票否决） | 40 | 识别成功 40 / 识别失败 0 |
| 用户价值举证 | 20 | 有明确、可验证的用户问题解决方案：20 / 只有业务诉求：10 / 只有感觉：0 |
| 和标准卡共存兼容性 | 15 | 同场景和标准卡并存不突兀：15 / 需要独立场景：8 / 不兼容：0 |
| 学习成本控制 | 10 | 无需学习：10 / 首次使用需提示：5 / 需要教学：0 |
| 视觉一致性 | 10 | 色彩、字体、圆角延续标准卡：10 / 部分偏离：5 / 完全另一套：0 |
| 数据指标友好度 | 5 | 对"有效曝光率 / CTR / 加购转化"有正向假设：5 / 持平：2 / 负向：0 |

**结论判定**：
- **≥ 80 分**：通过，纳入规范下一版（升级为正式分支，不再叫"变体"）
- **60~79 分**：**条件通过**，需在 1 个试点场景做 4 周 A/B 测试，用数据背书后再决定
- **< 60 分**：拒绝
- **跨场景同物种识别 = 0**：**直接拒绝**，分数再高也不过

## 五、豁免流程（对接 v4.0 "规范豁免流程"）

v4.0 文档第六节《风险与应对》中明确："规范与特殊业务需求冲突 → 建立规范豁免流程，逐案评审"。

### 豁免申请协议
业务方想要使用"不符合 variant-evaluation 评估"的变体，走豁免流程：

1. **申请**：业务对接人提交豁免申请书，必须包含：
   - 变体截图 + 使用场景
   - 不能走标准卡的业务理由
   - 预期用户价值 + 可衡量指标
   - 豁免期限（默认 ≤ 1 个迭代周期）
2. **评审**：评审委员会在 3 个工作日内裁决
3. **登记**：通过的豁免记入 `changelog.md` 的"豁免清单"章节，含过期时间
4. **复盘**：豁免期满必须复盘，按数据决定"转正（纳入规范）/ 回收（恢复标准卡）/ 续期"

### 豁免的上限
**同时活跃的豁免不超过 5 个**。超过上限，新的豁免申请一律拒绝，直到老的豁免关闭。**这个上限是为了防止"豁免成为规避规范的常态"**。

## 六、变体案例库（待沉淀）

每次评估的变体无论通过/拒绝都应沉淀到 `references/variants/` 目录（当前为占位，等 P1 阶段启动后开始积累）。

目录结构建议：
```
references/variants/
├── approved/      # 通过并转正的变体（含评分表 + 判定原因）
├── conditional/   # 条件通过，试点中的变体
├── rejected/      # 拒绝的变体（含拒绝原因，防止同类变体反复提报）
└── exempted/      # 豁免的变体（含过期时间）
```

案例库的价值：
- **给新设计师看**：什么样的创新能过，什么样的过不了
- **给评审看**：历史判例，确保一致性
- **给规范 owner 看**：哪些变体反复出现，可能需要升级成正式分支

## 七、评估协议的局限（诚实声明）

本协议**不能**解决：
- **业务判断**：该不该做这个变体（那是 PM 的活）
- **时机判断**：现在推出合不合适（那是运营的活）
- **数据判断**：变体实际表现（那是数据团队的活）

本协议**只能**回答一个问题：**如果这个变体上线，用户能否无障碍识别它仍然是一张内容双列卡？** 其余问题不在本 Skill 的能力边界内。
